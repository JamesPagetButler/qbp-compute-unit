April 2026 | Opus 4.6 (Red Team)

1. Executive Summary
The QBP Compute Unit is a Go-native inference engine for ternary (1.58-bit) language models that shares its core kernel with hypergraph traversal. A single assembly-accelerated function performs both model inference and associative memory retrieval. This evaluation recommends integrating it as a Walk-phase BRIDGE provider, handling lightweight inference tasks while llama.cpp handles heavy generative work. The routing decision is made by BRIDGE based on task complexity.
The benchmark gate passed in March 2026: Go’s inner loop is within 1.53x of C for ternary matmul. The 4.9x headline gap against C with SIMD is entirely attributable to vector instructions, closeable via Go Plan 9 assembly with no CGo dependency. The project proceeds as a Go program with one assembly-accelerated function — the standard Go pattern used by the stdlib’s crypto/aes and math/big.
The architectural insight: spreading activation on a typed hypergraph IS ternary matrix-vector multiply. Same inner loop, same assembly kernel, same data types. BMA doesn’t need two compute systems for inference and retrieval. It needs one system that operates in two modes depending on whether the matrix is dense (model weights) or sparse (graph adjacency). The compute unit is that system.

3. The Unified Engine: Inference + Retrieval
3.1 The Core Insight
A transformer forward pass is dense matrix-vector multiply: every weight participates in every token’s computation. Spreading activation on a hypergraph is sparse matrix-vector multiply: only the edges connected to the activated node participate. Both are multiply-accumulate (MAC) operations over packed low-bit values.

4.3 Routing Implementation
// In BRIDGE's ContextPreparer:

type TaskProfile struct {
    RequiresQuality   float64  // 0.0-1.0 (how much quality matters)
    RequiresSpeed     float64  // 0.0-1.0 (how fast it needs to be)
    GPUAvailable      bool     // is the GPU free?
    NetworkAvailable   bool    // is the internet up?
    UserFacing        bool     // will the user see this output directly?
    TokenBudget       int      // how many tokens expected
}

func (b *Bridge) SelectProvider(task TaskProfile) Provider {
    // Rule 1: Hypergraph retrieval always goes to compute unit
    if task.Type == TaskRetrieval {
        return b.computeUnit
    }

    // Rule 2: User-facing generation goes to best available
    if task.UserFacing {
        if task.GPUAvailable {
            return b.llamaCpp
        }
        if task.NetworkAvailable {
            return b.apiProvider
        }
        // Degraded: local ternary is better than nothing
        return b.computeUnit
    }

    // Rule 3: Background tasks prefer compute unit (no contention)
    if !task.UserFacing && task.RequiresQuality < 0.5 {
        return b.computeUnit
    }

    // Rule 4: Network down → local only
    if !task.NetworkAvailable {
        if task.GPUAvailable {
            return b.llamaCpp
        }
        return b.computeUnit
    }

    // Default: GPU if available, API otherwise
    if task.GPUAvailable {
        return b.llamaCpp
    }
    return b.apiProvider
}

4.4 The Possum State Advantage
When BMA enters Possum State (GPU consumed by gaming, thermal throttling), the compute unit keeps BMA alive. While llama.cpp inference is suspended and the GPU is unavailable:
Hypergraph retrieval continues. The compute unit runs on CPU. Spreading activation doesn’t touch the GPU. BMA can still search its memory.
FATHOM scoring continues. Observation classification runs on the 2B ternary model via CPU. BMA still observes the collaborator’s patterns even during Possum.
Sleep consolidation runs (CPU portion). The inner ring’s structural operations (type inference, contradiction detection, salience scoring) can use the compute unit. Only embedding recomputation (GPU) is deferred.
Draft generation is available. If the beekeeper sends a message through the reins during Possum, BMA can produce a response using the 2B ternary model. Lower quality than the 7–8B GPU model, but not silence.
The compute unit transforms Possum from “BMA is unconscious” to “BMA is drowsy but responsive.” The biological analog: the brain doesn’t shut down during sleep. It reduces activity, continues maintenance, and can still respond to stimuli.

6.1 The Honest Trade-Off
Building a Go-native inference engine from scratch is a significant engineering investment. The question is whether the unified engine (inference + retrieval in one kernel) justifies the cost compared to the simpler approach of just keeping llama.cpp for inference and HNSW for retrieval.
Arguments for the compute unit: Zero GPU contention for background work. BMA stays functional during Possum. Local inference with no network dependency. The assembly kernel serves both inference and retrieval (build one thing, get two capabilities). The octonionic edge structure, if it works, gives algebraically grounded knowledge representation. And the RISC-V path, if it materialises, means BMA eventually runs on silicon designed for its own cognitive architecture.
Arguments against: llama.cpp already works. HNSW already works. The compute unit adds complexity and engineering time for capabilities that BMA can live without during Walk. The quality gap between a 2B ternary model and a 7–8B quantized model is real and user-visible. The octonionic edge structure is theoretically elegant but unproven.
Recommendation: Build the assembly kernel and the BRIDGE provider interface. Use bitnet.cpp as a subprocess initially (same pattern as llama.cpp) to validate the routing logic. Replace with Go-native components incrementally as each one proves itself. This gets the architectural benefit (BRIDGE routing work to the right engine) without the upfront cost of a full Go inference implementation.

9.5 Width as a Scaling Function Toward Exactness
The width parameter is not bounded. QW8, QW16, QW32, QW64, QW128, QW256, QW512, QW1024 — the quaternion multiplication algorithm is identical at every width. The assembly kernel scales by processing more bytes per operand. The ceiling is memory and register width, not mathematics.
The scaling function: as width doubles, numerical precision roughly doubles in decimal digits. QW8 gives ~2 digits. QW32 gives ~9 digits. QW128 gives ~38 digits. QW512 gives ~154 digits. At QW1024, you have ~308 decimal digits of precision — territory where numerical computation approaches symbolic exactness. If a QBP algebraic identity needs verification to 300 decimal places, the same TMAC instruction handles it. Wider operands, more cycles, same operation.
This has architectural consequences for BMA:
Narrow widths are fast and cheap. QW8 traversal runs at 61 ns/edge. Graph retrieval, salience scoring, FATHOM observation classification — all run at maximum density with acceptable precision.
Wide widths are precise and expensive. QW64+ operations cost proportionally more cycles. Physics simulations, formal verification, and engineering calculations route to wider kernels. The cost is justified by the precision requirement.
BRIDGE never overspends precision. Every task carries a minimum PrecisionWidth. The compute unit selects the narrowest kernel that satisfies the requirement. A salience threshold check (QW8, <1 ns) never pays the cost of a cosine similarity (QW32, ~10 ns). The routing is automatic.
The same instruction, any width, any precision. On x86, wider widths map to multiple SIMD operations per quaternion element. On the RISC-V Run-phase, the custom TMAC/OMAC instruction natively supports a width parameter in the instruction encoding. On both architectures, the Go assembly file is the abstraction layer — the caller specifies the width, the kernel handles it.

[0,0,0,0,0,0,0,1] vs [0,0,0,0,0,0,0,-1] for the same nodes traversed in different order.
Significance: Octonionic edges encode the context of discovery, not just the fact of connection. “I found this concept through QBP theory” gives a different octonionic state than “I found this concept through Materia-Bio.” The graph records how you got there, not just what you found. This is exactly what FATHOM needs for modeling the collaborator’s epistemic path — the difference between knowing something and knowing how you came to know it.
This finding elevates octonionic edges from “elegant theory” to “functionally distinct from scalar.” The Walk Step 6 experiment should proceed with high confidence.
9.8 Gemini Experiment Results: Sleep Consolidation Durability
Gemini ran Experiment D: 1,000 simulated sleep cycles with salience decay, node merges, insertions, and deletions. The Quat8 graph remained numerically stable throughout, with average NormSq settling at ≈0.007. No NaN, no Inf, no edge weight collapse.
Conclusion: Quat8 edges do not require periodic renormalization. They are durable for the expected lifetime of a BMA instance across thousands of sleep cycles. This is a green light for Quat8 as the default edge representation from Crawl onward.
9.9 Gemini Experiment Results: Spreading Activation Requires Semantic Edges
Gemini ran Experiment C: spreading activation was 9,000× faster than linear search (2.93 µs vs 27.27 ms). However, with random edge topology, recall against embedding similarity was 0%.
This is expected and important. Spreading activation is not a replacement for HNSW with arbitrary edge weights. It is a complement that requires semantically meaningful edges. The fix: when a new hypergraph edge is created, initialize its weight from the embedding similarity between the connected nodes. Then spreading activation propagates along semantically weighted paths and retrieval quality should improve proportionally.
Crawl implementation requirement: Edge weight initialization from embedding similarity must be a Crawl-phase feature, not deferred to Walk. Without it, spreading activation produces fast but irrelevant results. With it, spreading activation becomes a 9,000× faster alternative to linear search with retrieval quality determined by edge weight quality.
