ROOT-CAUSE SUMMARY (read this first)
================================================================================

The "pure symbol rename" plan in the dispatch CANNOT WORK as written, for two
independent reasons in the existing repository state on origin/main:

  R1. emulator/asm/qmath_constants.s is in a SUBDIRECTORY of the emulator
      package. Go's `go build` only assembles .s files that sit alongside the
      package's .go files (i.e. inside emulator/). Files under emulator/asm/
      are NOT part of the emulator package; they are not assembled, not
      linked, and produce no symbols visible to the linker.

  R2. The generated symbols use file-scoped naming: `lean_sign_x<>(SB)`. In
      Plan-9 / Go assembly, the `<>` suffix marks a symbol as file-private
      (only visible within the .s file that defines it). Even if R1 were
      fixed by moving qmath_constants.s into emulator/, the `<>` would still
      hide lean_sign_x from qmath_amd64.s.

After deleting the four hand-derived DATA/GLOBL blocks per the dispatch and
renaming references to lean_sign_x<>, lean_sign_y<>, lean_sign_z<>, lean_conj<>,
the linker reports eight unresolved relocations (paste in §4 below) and the
test binary cannot be built. Therefore TestSIMDConstantsMatchROM,
TestDispatch_Equivalence, and the benchmark suite cannot run, so AC3, AC4,
AC5, AC6, AC7 are all unachievable on this attempt.

The dispatch context paragraph asserts:
  "The lean2rom pipeline (PR #12) generates emulator/asm/qmath_constants.s
   with symbols lean_sign_x, lean_sign_y, lean_sign_z, lean_conj. These are
   byte-equivalent to the hand-derived y_sign_x<> ... in emulator/qmath_amd64.s
   — proven by TestSIMDConstantsMatchROM."

That assertion is BYTE-equivalence-correct (the constants on origin/main do
have identical bit patterns), but the dispatch overlooks the LINKAGE
condition: byte-equivalence does not imply visibility. The lean_* symbols are
unreachable from the emulator package as the codebase exists on origin/main.

A correct attempt-2 strategy will need to do at least ONE of:
  (a) Define lean_sign_x<>, lean_sign_y<>, lean_sign_z<>, lean_conj<>
      DATA/GLOBL blocks INSIDE emulator/qmath_amd64.s itself (i.e. the rename
      keeps the constants file-local but renames y_sign_* → lean_sign_*).
      This satisfies the spirit of the migration (the symbol now signals
      "Lean-derived") but does not actually consume the generated file. The
      authority chain still rests on TestSIMDConstantsMatchROM.
  (b) Move emulator/asm/qmath_constants.s up to emulator/qmath_constants.s
      AND change every `lean_*<>` to `lean_*` (drop the file-scope marker)
      so the symbols are visible across .s files in the package. This
      requires editing emulator/asm/qmath_constants.s and/or moving it,
      which is outside the current SCOPE (and would also drift cmd/lean2rom
      output unless the generator is updated — explicitly forbidden by the
      dispatch).
  (c) Update cmd/lean2rom to emit the constants directly into
      emulator/qmath_constants.s (no `<>`), and rename the .s references
      accordingly. Forbidden by dispatch ("DO NOT extend lean2rom").

(a) is the only path that fits the current SCOPE. The reviewer should
re-dispatch with that direction or relax SCOPE to allow editing the
generator / moving the constants file.
