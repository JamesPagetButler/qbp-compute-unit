2. ARCHITECTURAL DEVIATIONS (RECORDED)
------------------------------------------------------------

Two deviations from doc/wyrd-integration.md §3.1 literal text. Both
documented in the package-level comment of public_api.go.

(a) Stateless `type Gearbox struct {}` not adopted: existing Gearbox
    in qword.go has internal state (ActiveWidth, big.Float scratchpads,
    QWord temporaries) consumed by cpu.go's ISA execution path. The
    new typed-per-width methods are added to the existing struct. From
    the caller's perspective they remain stateless — only QMulHighPrec
    transiently uses g.ActiveWidth + the existing big.Float pool, and
    that path snapshot/restores ActiveWidth so it doesn't disturb
    in-flight ISA execution.

(b) Width as iota not adopted: existing Width type in qword.go is
    `type Width int` with bit-count values (W64 = 64, W128 = 128, etc.)
    consumed by cpu.go and isa.go. Renumbering would break the ISA
    execution path. This file reuses the existing declarations
    unchanged. The doc's intent — "consumers must use named constants
    rather than literal integers" — is preserved.

These are the resolution the reviewer pre-specified in the dispatch.

The Go escape analysis is preserved because:
- Method receiver (*Gearbox) doesn't escape; methods don't read state
  on hot paths.
- Named-array conversions ([4]float64 -> QW64 -> [4]float64) are
  zero-cost (same memory layout, same size).
- The local `var dst QW64` stays on the stack across the qmul64
  pointer-passing convention because the underlying assembly kernels
  carry //go:noescape directives.
