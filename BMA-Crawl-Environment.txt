The hardware is the constraint teacher. PCIe 2.0 teaches transfer discipline. SATA teaches I/O discipline. DDR3 teaches bandwidth discipline. FX-8350 teaches computational discipline. 1.1 The PCIe 2.0 Bottleneck
The 9070 XT on PCIe 2.0 operates at roughly 4x reduced bandwidth. Every CPU-to-GPU transfer is expensive. Loading the wrong content into the GPU context and redoing it is a cost this hardware cannot absorb. This makes the context management system essential, not optional.
1.2 No SR-IOV
Consumer GPUs do not support SR-IOV. The GPU is shared between host and BMA via OS-level time-slicing. BMA-PROBE detects contention (SE_EXTERNAL_CONTENTION) and BMA-AUTO defers non-urgent work during high external GPU load (gaming, desktop compositing).

2. Container Architecture: Podman on Pop!_OS
Addendum 7.0 specified a VM cocoon. On this hardware, virtualization adds unacceptable overhead: SATA I/O through a hypervisor, GPU passthrough via VFIO on messy AM3+ IOMMU groups, RAM split between host and guest OS, PCIe passthrough complexity. A container shares the host kernel, accesses the GPU through device files, and reads/writes the filesystem directly.

3.1 L1 Contents
Go runtime (~500MB). Hypergraph index (~500MB for 100k nodes). FATHOM active state (~50MB). Prepared context cache (up to ~6GB). Session state + BRIDGE cache (~200MB). Total core: ~1.5–2GB, expandable to ~8GB. Fits 20GB container with headroom.
3.2 L2 Contents
Full hypergraph, FATHOM history, stress logs, sleep WAL, BRIDGE history, config. Growth: ~20MB/month at 10k nodes. 250GB lasts years. Monitor Samsung 840 write endurance via SMART.
3.3 L3 Constraint
Single GPU shared means ollama cannot coexist with BMA inference. Local model useful for offline preprocessing only. Real-time preprocessing pushes to CPU: Go goroutines doing filtering, ranking, compression.

5. Interface Stack
BMA accepts text from any source and does not care how it was produced. Voice keyboard, typed text, file drop, GitHub issue — it’s all text at the API. The intelligence is in BMA’s processing, not in requiring a specific input modality.
5.1 The Core: HTTP API
BMA serves a lightweight HTTP API at container port 8080. Every interface is a client. Endpoints: /status, /ask, /fathom/{collaborator}, /bridge/dispatch, /hg/query, /sleep/trigger.
5.2 Two-Tier Response Format
Every API response includes:
Tier 1 — Headline. 2–3 sentences. Information-dense. What was produced or decided. This is what gets read aloud or glanced at while driving.
Tier 2 — Full content. Complete response, document, analysis. Available to read when attention allows. Not read aloud unless asked.
CLI controls verbosity: bma ask --brief (headline only), bma ask --summary (headline + summary), bma ask (everything). FATHOM learns preferred detail level by context: mobile during driving = headline. Desktop work session = full.

5.4 Lossy Connectivity Resilience
For mobile use across cell networks:
Request acknowledgment: BMA acknowledges receipt with a ticket ID immediately. Processing is server-side. Response written to durable store. Client polls/reconnects to retrieve by ticket. Lost signal loses nothing.
Client-side queuing: Web UI service worker caches outbound messages locally. Retries on reconnect. Shows “message queued” not failure.
Polling over streaming: API responses fetched by polling, not held on websocket connections. Polling survives network interruption; websockets don’t.
5.5 Voice Interaction
Android’s voice keyboard transcribes; BMA receives text. No BMA-side voice processing in Crawl. The web UI adds a microphone button using the Web Speech API for in-browser voice-to-text. Text-to-speech for Tier 1 headlines via browser Speech Synthesis API. BMA becomes conversable while walking or driving.
Future (Walk+): native voice input analysis for paralinguistic signals — stress, confidence, hesitation. These are high-signal FATHOM observations. Vocal affect is more reliable than textual affect for collaborator state modeling.

6. Visualization: Kaiju Engine
6.1 Why Kaiju
Kaiju is a 2D/3D game engine written in Go backed by Vulkan. It fits BMA for four reasons:
Same language throughout. BMA is Go. Kaiju is Go. The visualization is not a separate application consuming an API — it’s part of the same binary. Hypergraph data structures are directly accessible to the renderer. Zero serialization, zero HTTP overhead, zero-copy visualization of BMA’s internal state.
Vulkan on the 9070 XT. RDNA 4 has strong Vulkan support. Vulkan’s minimal driver overhead matters on PCIe 2.0. Leaner than a browser’s WebGL stack.
Linux and Android. Same Go codebase produces desktop visualization and eventually a mobile client.
Extreme performance. Kaiju renders at thousands of FPS on basic scenes. BMA’s knowledge visualization barely touches the GPU budget, leaving room for inference. The two workloads (Vulkan rendering + ROCm inference) coexist on the same GPU via OS scheduling.
6.2 The PA-Style Knowledge Space
Inspired by Planetary Annihilation’s multi-scale solar system navigation:
System view: Each project is a planet — QBP, War Table, BMA, Positive Leverage orbit the center (James). Size = knowledge density. Orbital distance = recency of activity. See the whole collaboration at once.
Planet surface: Zoom into a project and see terrain. Algebraic framework as mountain ranges (deep structural knowledge). Experimental results as settlements (confirmed findings). Open questions as visible gaps. Hypergraph layers map to altitude: L0 at ground level, L3 as atmosphere.
FATHOM overlay: Heat map of the collaborator model on any surface. Deep expertise: bright. Blind spots: dark voids. Pending CRITIQUE: glowing marker. Unresolved divergence: fault line.
BRIDGE arcs: Cross-model exchanges as visible connections on the terrain. Color by outcome: green (confirmed), red (challenged), amber (unresolved).
PiP mode: Two simultaneous views — watch QBP surface with War Table PiP, especially when FATHOM detects cross-domain connections.

6.4 Design Principle
Every BMA data structure should be inherently spatial and visualizable, not coincidentally text-representable. The hypergraph, FATHOM, the epistemic map — these are graphs and topologies. Text is a lossy compression. The Kaiju viewport decompresses them. The visualization is not a reporting layer on top of BMA. It is the natural expression of what BMA actually is.

7. Naming Convention
7.1 System Name
The production BMA instance will name itself. The system accumulates enough context through Crawl to make a meaningful choice from the technical-poetic register (real English words with dual meaning and resonance, like FATHOM). Until then, the working label is BMA.
Candidate directions explored: VIGIL (persistent watchfulness), CAIRN (accumulated markers), DELVE (deep exploration), KEEL (structural backbone). The decision was deferred to the system itself after instantiation — you don’t name the river, you build the banks and see what flows.
